# Snap Grid Spec

## Idea

Add a page setting that lets users enable snap-to-grid, with a default grid size and direct user control over the size.

## HCI/UX Research Basis

- Nielsen Norman Group describes drag and drop as direct manipulation useful for moving and resizing objects, but it must be designed carefully.
- Material Design layout guidance treats layout systems as tools for predictable, adaptable placement.
- Nielsen Norman Group's visibility-of-system-status heuristic supports showing when snapping is active.
- WAI-ARIA keyboard guidance implies the grid feature should have keyboard alternatives, not drag-only behavior.

Sources:
- https://www.nngroup.com/articles/drag-drop/
- https://m3.material.io/foundations/layout/layout-overview/parts-of-layout
- https://www.nngroup.com/articles/ten-usability-heuristics/
- https://www.w3.org/WAI/ARIA/apg/practices/keyboard-interface/

## User Experience

Snap grid should help precision without surprising users:

- Page styles/settings include `Snap to grid` toggle.
- Default grid size is `16px`.
- Users can set grid size directly.
- Optional `Show grid` toggle renders a light background grid.
- While dragging, the Area should preview its snapped position.
- Holding a modifier key, such as Option/Alt, temporarily disables snapping.

## Data Model

```ts
type SnapGridSettings = {
  enabled: boolean
  visible: boolean
  size: number
}
```

Store this under page settings, not per Area.

## Interaction Rules

- When enabled, Area move and resize endpoints snap to the nearest grid increment.
- Existing Areas do not jump when the setting is enabled; snapping applies on the next move/resize.
- Grid size must be clamped to a safe range, such as 4-128 pixels.
- Keyboard movement can use arrow keys for 1px nudges and Shift+Arrow for grid-sized nudges after keyboard movement exists.

## Acceptance Criteria

- User can toggle snap grid from Page styles or Settings.
- User can set grid size directly.
- Dragging an Area snaps its final position to grid increments.
- Holding the bypass modifier allows free movement.
- Grid visibility is light enough not to compete with content.
- Grid settings persist in page JSON.
- Tests cover snap rounding and grid-size validation.

## Open Questions

- Should resize snap width/height, x/y edges, or both?
- Should grid visibility default to off when snapping is on?
- Should the grid origin be page origin or configurable?
